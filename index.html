AI → ML → Deep Learning → GenAI Hierarchy
  • AI (Artificial Intelligence) — Broadest field — any machine that simulates human intelligence
  • ML (Machine Learning) — Subset of AI — machines learn patterns from data without explicit rules
  • Deep Learning — Subset of ML — uses multi-layered neural networks, excels at images/audio/text

Exam trigger "Which is a subset of which?" — draw the nesting: AI → ML → Deep Learning → GenAI. Every inner field IS a type of the outer one.
Supervised / Unsupervised / Reinforcement Learning
  • Supervised — Labelled data (input + correct answer). Learns to predict. Used for classification (categories) and regression (numbers)
  • Unsupervised — Unlabelled data (no correct answer). Finds hidden patterns. Used for clustering and anomaly detection
  • Reinforcement — No labelled data. Agent learns by trial and error — receives rewards for good actions, penalties for bad ones
Exam trigger "What type of ML?" — ask: Is data labelled? Yes → Supervised. No labels, find groups → Unsupervised. Learning by doing/reward → Reinforcement.
ML Pipeline — All 9 Stages
  • 1. Data Collection — Gather raw data from databases, APIs, sensors, or files
  • 2. EDA (Exploratory Data Analysis) — Explore data to find patterns, outliers, distributions — understand what you have before cleaning
  • 3. Data Pre-processing — Clean data: remove nulls, fix errors, normalise values, encode categories
  • 4. Feature Engineering — Create new meaningful input variables from existing data (e.g. "days since last purchase")
  • 5. Model Training — Feed clean, engineered data into the algorithm so it learns the pattern
  • 6. Hyperparameter Tuning — Adjust model configuration settings (learning rate, layers) to improve performance
  • 7. Model Evaluation — Measure performance on held-out test data using metrics (Accuracy, F1, AUC-ROC)
  • 8. Deployment — Make the model available for predictions via API endpoint or batch job
  • 9. Monitoring — Watch deployed model for drift — alert and re-train when accuracy degrades
Exam trigger "Which stage?" — ordering questions follow this exact sequence. Cannot train before collecting and cleaning data. Cannot deploy before evaluating.

Model Evaluation Metrics (Performance + Business)
Technical metrics (measure model quality)
  • Accuracy — Percentage of predictions that are correct. Misleading on imbalanced datasets
  • F1 Score — Balances precision and recall — best for imbalanced data (fraud, disease detection)
  • AUC-ROC — Measures how well the model separates classes (0.5 = random, 1.0 = perfect)
Business metrics (measure real-world impact)
  • ROI — Return on investment — did the model generate more value than it cost?
  • Cost per user — Inference cost divided by number of users served
  • Customer feedback — Qualitative measure of whether outputs meet user expectations
Exam trigger "Which metric is most appropriate?" — imbalanced dataset → F1 or AUC-ROC. Balanced dataset → Accuracy. Business impact question → ROI or cost per user.
When to Use vs When NOT to Use AI/ML
Use AI/ML when:
  • Decision-making at scale — Millions of decisions per day that no human team could handle (fraud checks, recommendations)
  • Complex pattern recognition — The pattern is too complex to write as a rule (image recognition, NLP)
  • Scalability needed — Workload grows faster than you can hire people
Do NOT use AI/ML when:
  • A simple rule already works — Tax = price × 0.08 is an exact formula — ML adds cost and complexity with no benefit
  • Guaranteed outcome required — ML is probabilistic — if you need a deterministic result, use traditional programming
  • Insufficient data — ML needs enough labelled examples to learn a meaningful pattern
  • Cost outweighs benefit — Compute and development costs exceed the value of the improvement
Exam trigger "Should they use ML?" — if the answer can be expressed as a formula or lookup table, the answer is NO.

Overfitting, Underfitting & Bias-Variance Tradeoff
Diagnose first:
  • High training accuracy + low test accuracy — Overfitting (high variance) — model memorised training data
  • Low accuracy on BOTH training and test — Underfitting (high bias) — model too simple
6 solutions for overfitting:
  • More training data — Harder to memorise all patterns
  • Regularisation (L1/L2) — Penalises model complexity
  • Cross-validation (k-fold) — Tests generalisation across multiple splits
  • Simplify model / reduce features — Removes noise the model memorises
  • Dropout — Neural networks only — randomly disables neurons
  • Early stopping — Stop when validation loss starts rising
Fix for underfitting:
  • Add more features, increase model complexity, train longer —
Exam trigger "Performs well on training, fails on new data" → overfitting. "Poor on both" → underfitting. "Must document which variable drives the decision" → use Decision Tree not Neural Network.
Train / Validation / Test Split and Data Leakage
  • Training set — Model learns the pattern from this data
  • Validation set — Tune hyperparameters and monitor overfitting DURING development — used repeatedly
  • Test set — Final one-time evaluation AFTER all development is complete — never seen during training or tuning
  • Data leakage — When information from outside the training set contaminates the model. Cause: falsely high development accuracy that collapses in production
Exam trigger "Evaluate final model" → test set. "Tune hyperparameters" → validation set. "Perfect dev accuracy + production failure" → data leakage.
Example

Including a post-event field (fraud investigation outcome) in training data looks accurate in development but that field does not exist at prediction time — classic leakage.

Neural Networks & Deep Learning
  • Neural network — Computing system of interconnected nodes (neurons) organised in layers — inspired by the human brain
  • Layers — Input layer (receives data) → Hidden layers (extract features) → Output layer (produces prediction)
  • Deep learning — Neural network with many hidden layers — "deep" = many layers. Learns complex features automatically
  • Best for — Images, audio, text, video — unstructured data where features are hard to hand-engineer
Exam trigger "Image recognition" / "speech recognition" / "language understanding" → deep learning / neural network.
Data Types in AI
  • Structured — Rows and columns — databases, spreadsheets, CSV files. Easy to query and analyse
  • Unstructured — Text, images, audio, video — no predefined format. Requires ML to extract meaning
  • Semi-structured — Has some organisation but not rigid rows/columns — JSON, XML, emails
  • Tabular — Structured data in table format — the most common ML input type
  • Time-series — Data points ordered by timestamp — stock prices, sensor readings, sales over time
  • Labelled — Each data point has a known correct answer (used for supervised learning)
  • Unlabelled — No correct answer assigned (used for unsupervised learning)
Exam trigger "What type of data?" — timestamps + ordered → time-series. No rows/columns → unstructured. Rows/columns → structured/tabular.
Classification vs Regression
  • Classification — Output is a category label. Examples: spam/not spam, approve/deny, type A/B/C. Can be binary (2 classes) or multiclass (3+ classes)
  • Regression — Output is a continuous number. Examples: house price, sales forecast, temperature prediction
  • Key rule — Look at the output type first. Category → Classification. Number → Regression
Exam trigger "Which ML type?" — always check the output. "Classify genes into 20 categories" → multiclass classification. "Predict next month revenue" → regression.
Batch vs Real-time Inferencing
  • Batch inferencing — Process large volumes of inputs together in one run. Results are delayed (minutes to hours). High throughput, lower cost per prediction
  • Real-time inferencing — Instant prediction on a single input as it arrives. Results in milliseconds. Required for time-sensitive decisions
Exam trigger "Instant decision at point of transaction" / "fraud check at checkout" → Real-time. "Process 1M records overnight" / "monthly scoring run" → Batch.
Model Sources: Open-Source vs Custom
  • Open-source pre-trained — Publicly available models (Llama, Mistral) deployable via SageMaker JumpStart. Fast start, low initial cost, less control
  • Proprietary managed — FMs accessed via API (Bedrock — Claude, Titan, Nova). No infrastructure, serverless, pay per token
  • Custom trained — Built from scratch on your own data. Most tailored, most expensive, most control. Rarely needed for AIF-C01 scenarios
Exam trigger "Low budget + quick start" → open-source via JumpStart. "No infrastructure" → Bedrock managed API. "Highly proprietary domain" → custom training.
Deployment: Managed API vs Self-Hosted
  • Managed API (Bedrock / SageMaker endpoints) — AWS hosts and runs the model. No server management. Scale automatically. Pay per use
  • Self-hosted (EC2 / ECS / EKS) — You deploy the model on your own compute. Full control over environment, networking, model version
Exam trigger "Minimal operational overhead" / "no infrastructure management" → Managed API. "Full control" / "private VPC only" / "custom networking" → Self-hosted.
MLOps Fundamentals
  • Experimentation — Track and compare different model runs, hyperparameters, and results systematically
  • Repeatable processes — Same pipeline produces same results every run — no manual steps
  • Model monitoring — Watch deployed models for accuracy degradation over time (model drift)
  • Model re-training — Trigger re-training when drift is detected — update model with recent data
  • Scalable systems — Pipeline handles growing data volumes without manual intervention
Exam trigger "Which MLOps practice?" — "tracking experiments" → Experimentation. "Performance dropping in production" → Model Monitor + re-training. "Same result every run" → Repeatable processes.
AI Adds Value: 3 Core Reasons
  • Decision-making at scale — AI makes millions of consistent decisions per day — fraud checks, content recommendations, risk scoring
  • Automation of complex patterns — Replaces repetitive tasks that require recognising patterns humans find hard to define as rules
  • Scalability — Handles growing workloads without proportionally increasing headcount or cost
Exam trigger "Why use AI here?" — match the scenario to one of these three: volume of decisions, pattern complexity, or growth that cannot scale with humans.

GenAI Advantages vs Disadvantages
Advantages (3 named in exam guide):
  • Adaptability — Works across many different tasks without task-specific training
  • Responsiveness — Answers in real time — instant response to queries
  • Simplicity — Non-technical users can use it with natural language — no ML expertise needed
Disadvantages (4 named in exam guide):
  • Hallucination — Generates confident but factually incorrect information
  • Interpretability — Hard to explain why the model produced a specific output
  • Inaccuracy — Outputs are not always correct — cannot be fully trusted without verification
  • Nondeterminism — Same prompt can produce different outputs each time
Exam trigger "Which is a benefit/limitation of GenAI?" — match the option to the official list. Adaptability/Responsiveness/Simplicity = advantages. Hallucination/Interpretability/Inaccuracy/Nondeterminism = disadvantages.
Token / Chunking / Embedding / Vector — Core GenAI Terms
  • Token — The smallest unit an LLM processes (~0.75 words). LLMs read tokens, not characters or whole words
  • Context window — Maximum tokens the model can hold in memory at once. Exceeded → model forgets earlier content
  • Chunking — Splitting large documents into smaller pieces so they fit within context window for RAG storage
  • Embedding — A list of numbers that represents the meaning of text — similar meanings = nearby positions
  • Vector — The numerical representation (embedding) stored in a vector database for similarity search
Exam trigger "How does RAG store documents?" → chunk → embed → store as vector. "Why does model forget earlier conversation?" → context window limit.
Cost Tradeoffs: Token Pricing vs Provisioned Throughput
  • Token-based pricing — Pay per token processed (input + output). No commitment. Best for variable, unpredictable workloads
  • Provisioned throughput — Reserve dedicated model capacity at a fixed cost. Guarantees consistent response speed. Best for high-volume, consistent workloads
Exam trigger "Variable / unpredictable usage" → token pricing. "Guaranteed response time" + "high consistent volume" → provisioned throughput.
Business Value Metrics for GenAI Applications

  • ARPU — Average Revenue Per User — total revenue divided by active users over a period
  • CLV — Customer Lifetime Value — total revenue from a customer over their entire relationship
  • Efficiency — Time or cost saved by the AI compared to the manual process
  • Accuracy — Percentage of AI outputs that are correct — task-specific
  • Cross-domain performance — How well the model works across different subject areas simultaneously

FM Customisation: Which Approach When?
  • Prompt engineering — Craft better instructions. No training, no model changes, no cost. Use first
  • In-context learning — Include examples or instructions in the prompt (zero/single/few-shot). Same as prompt engineering — no training
  • RAG — Retrieve external documents at query time. Best for fresh/private data. Reduces hallucination
  • Fine-tuning — Retrain model weights on domain data. Best for specific style, tone, or vocabulary
  • Continuous pre-training — Extend base pre-training on large domain corpus. Best for teaching an entirely new domain
  • RLHF — Human raters score outputs; model learns from preferences. Most expensive. Best for value alignment
Exam trigger "Which approach?" — private/fresh data → RAG. Specific writing style → Fine-tuning. New domain from scratch → Continuous pre-training. Quick test → Prompt engineering. Cost order: prompt < RAG < fine-tuning < RLHF.

RAG Architecture — How it Works
Setup (done once):
  • 1. Chunk — Split large documents into smaller sections
  • 2. Embed — Convert each chunk into a numerical embedding vector
  • 3. Store — Save embeddings in a vector database (OpenSearch, Aurora pgvector, etc.)
Runtime (every query):
  • 4. Query arrives — User submits a question
  • 5. Retrieve — Embed the query and find the most similar chunks in the vector DB
  • 6. Generate — Inject retrieved chunks into the FM prompt — FM generates a grounded answer
Exam trigger "Reduce hallucination using company documents" / "keep AI answers current without retraining" / "private data + FM answer" → RAG.
Prompt Engineering — All Techniques + All Risks
Techniques:
  • Zero-shot — No examples — ask directly
  • Single/few-shot — 1 or 2–5 examples in the prompt to guide format
  • Chain-of-thought — "Think step by step" — improves complex reasoning
  • System prompt — Sets model behaviour, persona, and constraints for all responses
  • Negative prompt — Tells model what to exclude
Risks:
  • Injection — Hidden malicious commands in user input
  • Jailbreaking — Bypass safety via roleplay/persona
  • Poisoning — Corrupt content in RAG source data
  • Exposure — Model reveals system prompt or secrets

Responsible Data: Dataset Requirements & Bias Effects
What a responsible dataset needs:
  • Inclusivity — Represents all relevant demographic groups (age, gender, ethnicity, language)
  • Diversity — Wide variety of examples and perspectives
  • Curated sources — Carefully vetted, high-quality data — not random web scraping
  • Balance — Roughly equal representation across all classes and groups
What happens when data is biased:
  • Skewed data → unfair outcomes — Model performs better for over-represented groups, worse for under-represented ones
  • Overfitting bias (high variance) — Model memorises group-specific noise — fails on unseen demographic subgroups
  • Underfitting bias (high bias) — Model too simple — systematically wrong across all groups
Exam trigger "What makes a dataset responsible?" → inclusivity, diversity, curated, balanced. "Model performs well for group A, poorly for group B" → data bias → use SageMaker Clarify + subgroup analysis.
Explainability in Practice

Making explanations useful (Human-Centered Design):

Transparency tools:

Algorithm Interpretability — One-Line Exam Rules

How to use: Strip the business story. Extract the two constraints. Match to the rule below. The answer is always the algorithm that satisfies BOTH constraints simultaneously.
Decision Tree
Multiple categories + must trace/document each step
A judge reading every ruling aloud, clause by clause. Each decision node is spoken out loud in court.
Why others fail: Random Forest = jury verdict only, cannot trace individual reasoning. Neural Network = black box machine, no trace possible.
Logistic Regression
Binary outcome (yes/no) + must show which factors influenced the decision
A doctor's whiteboard: approval score = income×0.4 + credit×0.3 + employment×0.3. Every coefficient visible.
Why not Decision Tree: Binary output + linear factors → Logistic Regression. Decision Tree is better for multiple categories and non-linear splits.
Naive Bayes
Probability of each class must be visible + simple model acceptable
A weather forecaster: "70% rain because humidity=50%, clouds=30%, wind=20%." Probabilities per class shown explicitly.
Why not others: Only Naive Bayes shows probability per class as its natural output. Common for text classification and simple probabilistic tasks.
Neural Network
Maximum accuracy only — interpretability not required or mentioned
A black box machine. Data goes in, answer comes out. Nobody sees what happens inside. Use when accuracy is the only constraint.
Why not others: If the exam says "must explain" → eliminate Neural Network immediately. Neural Network wins ONLY when accuracy is the sole requirement.
