LLM
AI Concepts

An LLM — a Large Language Model — is trained on a massive amount of text to recognize patterns and generate language. It can handle many tasks, but it has four limitations that drive every product decision.

Hallucination — the model produces confident output that is factually wrong, and it does not know it. Knowledge cutoff — blind after its training date. No memory — each call starts fresh. Context window — there is a hard limit on text per call, so long documents must be chunked.

On cost — a token is about three-quarters of a word and you pay per token. Ten thousand notes at two thousand tokens each is twenty million tokens a day. Always calculate that before committing. Bigger models are more accurate but slower and more expensive. I define the requirements. Engineering picks the model.

NLP vs LLM
AI Concepts

NLP tools are specialists — built for one task, fast, cheap, and fully explainable. LLMs are generalists — flexible and fluent, but they can hallucinate and cost more to run.

The rule is simple. If you need to extract a specific fact — a billing code, a diagnosis — use NLP. If you need to generate natural text — a session note, a summary — use an LLM. NLP extracts the codes. The LLM writes the notes.

I would not default to an LLM just because it sounds more advanced. An NLP tool is more auditable — you can trace exactly why it gave the answer it did. That matters a lot in a regulated environment.

RAG
AI Concepts

RAG — Retrieval Augmented Generation — gives the model access to real documents at query time instead of relying on frozen training memory. Text is converted into numbers called embeddings, a vector database finds the most relevant documents by meaning — that is semantic search — and the model uses those documents to generate its answer.

Every answer traces back to a real document. That means it is auditable. And it stays current without retraining — which matters a lot when guidelines and codes change regularly.

If RAG gives poor answers, the problem is usually embedding quality or chunking strategy — not the model. Chunk size, overlap, and embedding model choice are product requirements I define, not engineering details.

Prompt Eng → RAG → Fine-tuning
AI Concepts

There are three ways to improve a model and you always try them in order — never skip steps.

Prompt engineering first — improve the instructions you give the model. Free, takes hours, the model does not change. Always try this first. Then RAG — give the model access to real documents. Every answer traces to a source. Takes days to weeks. Use this for policies and guidelines that change often. Then fine-tuning — continue training on your own data. The model changes permanently, takes weeks, and is expensive. Only when everything else has failed.

Recommending fine-tuning before exhausting the first two options signals poor product judgment. It is the most costly path and usually unnecessary.

Training vs Inference
AI Concepts

Training is when the model learns from data — slow, expensive, happens every few months. Inference is when the model runs in production — fast, real-time, thousands of times a day.

As TPO, I live in the inference world. I define the latency requirement, the cost per call, the fallback behavior, and I make sure drift monitoring is in place. Training is a roadmap decision — I trigger it based on what I see in production.

I would retrain when user acceptance drops below the alert threshold or when there is a significant change in the data the model needs to understand. Monitoring is how I know when that moment has arrived.

ICD Codes
AI Concepts

ICD codes — International Classification of Diseases — are standardized numbers used to label diagnoses. For example, F32.1 means Major Depressive Disorder, moderate. They are used in billing, records, and reporting.

The key issue is that ICD codes change with each version release. If you rely on a model's frozen training memory, it will not know about new codes after the cutoff date.

That is why I would always push for RAG for anything involving codes — it pulls from a live, updated document library. I would define knowledge base freshness as a product requirement and put a refresh schedule and a freshness SLA on the roadmap.

Hallucination
Failure Modes

The model can produce confident output that is completely wrong — and it does not know it. That is the problem with hallucination. In a healthcare product, that is a patient safety risk. An invented dosage or a fabricated diagnosis could directly harm someone.

The way I would handle it — four protections. A review gate so no clinical output reaches a record without approval. RAG grounding to anchor the model in real documents. Continuous monitoring of the hallucination rate in production. And a pre-defined rollback trigger — if the rate crosses the threshold, the feature turns off immediately.

The distinction worth making — hallucination is wrong from day one, so you need a review gate. Drift is correct at launch but gets worse over time, so you need monitoring. Different problem, different fix.

Model Drift
Failure Modes

Model drift is when the model gets worse over time — not because the model changed, but because the world did. New codes, updated guidelines, evolving language. The model was trained on old data and slowly falls behind.

Three responses. Record baseline metrics at launch — that is the reference point. Set an alert threshold — if acceptance drops below 75%, the team investigates. Put retraining on the roadmap as a recurring scheduled item, not a crisis response.

Monitoring must ship with the feature. Without it, drift fails silently for months. The distinction — drift is correct at launch, then degrades. Hallucination is wrong from day one.

Precision / Recall / F1
Metrics

Precision is about false alarms. When the model flags something — how often is it actually right? If it flags ten things and only eight are real, that is 80% precision. The other two were wasted effort.

Recall is about what gets missed. Out of all the real problems that exist, how many did the model catch? If there are ten real problems and the model only found seven, that is 70% recall. Three slipped through. F1 score balances both into one number.

In healthcare, recall matters more. Missing a real case is worse than a false alarm. So if an engineer tells me recall is low, I say — we are missing real cases. That is not acceptable. I decide what error level is okay. Engineering figures out how to get there.

p95 Latency + Tradeoffs
Metrics

p95 latency means 95 out of 100 requests respond within the target time. The slowest 5% may take longer — that is acceptable. It gives a precise, measurable guarantee.

In acceptance criteria I always write — must respond within 2 seconds at p95. Never "must be fast." That is not measurable and leads to disagreements at the end of a sprint.

On tradeoffs — for a real-time feature, optimize for speed. For an overnight batch process, optimize for accuracy. My job is to define the requirements. Engineering picks the model. Always match the model to the use case.

AC Rules + Fallback
Product Process

For AI features, I follow four rules when writing acceptance criteria.

Rule one — never write "the output is correct." AI is not always right, so I write a number instead. For example — the note is accepted without major edits in 85% or more of cases.

Rule two — agree upfront on what each performance level means. Above 85%, no action needed. Between 75 and 85%, the team investigates but the feature stays live. Below 60%, the feature turns off immediately.

Rule three — always define what happens when the model is not confident. Show a blank form. Never show a low-quality AI output to a user.

Rule four — monitoring goes live on the same day as the feature. Not next sprint.

A code bug fails loudly. A bad AI model fails quietly for months — unless monitoring ships with the feature.

DoD Checklist
Product Process

Before any AI feature ships, I check five things.

The eval report must hit the exact number we agreed on — not close, the exact number. Clinician sign-off — a real clinician reviewed a sample and approved it, QA alone is not enough. Monitoring must be live on launch day, not next sprint.

The fallback path must be actually tested — when the model is not confident, it shows a blank form, not a bad output. And HIPAA must be cleared — a signed agreement in place for any external vendor that touches patient data. If any one of these is missing, the feature does not ship.

HIPAA / AEBA
Compliance

Four checks before any feature that touches patient data ships. Memory hook — A, E, B, A.

A — Authorization. Write RBAC rules in the acceptance criteria. A billing admin cannot access therapy notes. Users see only what their role permits.

E — Encryption. In transit — TLS 1.2. At rest — AES-256. Unencrypted PHI is a HIPAA violation.

B — BAA — Business Associate Agreement. Required before PHI touches any external API. No BAA means a violation. Non-negotiable.

A — Audit Logs. Every PHI access logged — who, when, why. Required for compliance audits. Belongs in the Definition of Done.

A · E · B · A — all four or it does not ship.

PHI + RBAC
Compliance

PHI — Protected Health Information — is any data that can identify a patient. Names, session notes, diagnosis codes, billing records, dates of service. If it links back to a real person, it is PHI and it must be protected.

RBAC stands for Role-Based Access Control. It means each user can only see and do what their role allows. A billing admin can access financial records but not therapy notes. A therapist sees their own clients only, not another therapist's clients.

I write these access rules explicitly in the acceptance criteria. I never assume the default settings are correct — in healthcare, a wrong default can be a compliance violation.
